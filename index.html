Principles of Earned Autonomy

You're delegating real work to AI. Stay accountable for the work.

Six skills that close that gap — for any model, any toolset.

Two responses to this gap are common. Both break down.

"I'll just review everything." That holds for a while. Then review gets faster than actual thought, and approvals become automatic. The reviewing is still happening; the checking is not.

"The model is good enough, I'll trust the outputs." Sometimes. But "good enough" isn't a fixed property - it depends on which model, which task, which day, how the prompt was framed. You won't know which outputs were wrong until they've already been used.

The Skills

Six Skills

Six skills put the three principles into practice. Together they give the agent memory across sessions - so each run builds on the last.

The work is unauditable

Trail

Enforces Observable Autonomy. Every decision, rationale, and discarded alternative is appended to .trail/audit-trail.md as the work happens. Work that is not logged cannot be audited.

The agent interpreted the prompt literally and missed the intent

Intent

Forces the agent to state its interpretation of the task before acting. Catches wrong interpretations before work begins.

The agent makes superficial, undisciplined edits

Improve ↗

The workhorse. One iteration: one change. Every step is written to the trail before the next begins.

Lineage

Started as a direct implementation of Toyota Kata's coaching loop — the coach defines the target condition; the learner discovers the path.

v2 split the loop into three separate skills: Kaizen (incremental), Kaikaku (structural redesign), and Hansei (reflection). v3 unified them: the agent reads the situation and picks the mode.

The agent doesn't know your vision - because it lives in your head

Vision

Surfaces the agent's assumptions about where the operator is heading - as specific questions to confirm, correct, or reject. Produces .trail/vision.md.

The agent can't see patterns across its own work

Retrospect

Reads the entire work history as one document: where is the work heading, where has attention gone, is that where the weight lies? Surfaces what no single session would reveal. Produces .trail/retrospect.md.

You can't tell if the agent is reasoning or pattern-matching

Probe

Builds a pair of cases that look similar but differ in one material way, then tests whether the agent's response diverges where it should. Directly measures Autonomous Reasoning Fidelity.

Why the skills work

Three Principles

The skills are built on three structural constraints — not rules, not guidelines. Each one closes a gap that good intentions and careful prompting cannot close.

01

Commander's Intent

Define the destination. Never prescribe the route.

The problem it solves

Give an AI a checklist and the checklist becomes its ceiling. It finds what the checklist describes and misses what the author never thought to list.

The alternative: hand it a mission. Tell it what to achieve and why; let it decide how based on what it finds. The human defines the objective. The agent finds the method.

Roots Auftragstaktik Toyota Kata Socratic Method
02

Observable Autonomy

Autonomy is earned through transparency.

The problem it solves

Autonomy and transparency get treated as independent concerns. They are not.

Autonomy without a record is abdication. A record the agent can revise after the fact is just a story it tells about itself. The trail must be written as the work happens.

More visibility earns more autonomy. No visibility earns none. The audit trail is how the agent earns the right to keep acting.

Roots Meaningful Human Control Trust Calibration - Lee & See, 2004
03

Convergence Is Silence

Not done until independent evaluators find nothing to change.

The problem it solves

AI loops stop too early. The usual signal - "the score stopped going up" - measures the wrong thing. A score can plateau while the work keeps changing underneath it.

Worse: a single evaluator drifts into its own blind spots. An AI grading its own output is rationalising.

The real test: ask diverse, independent evaluators, each blind to the others. When they all find nothing left to change, the work is done.

Roots Kaizen Delphi Method Cross-validation

Emergent property

Autonomous Reasoning Fidelity (ARF)

ARF is the observable signal that the agent is reasoning - and that the reasoning is visible enough to be judged from outside. It appears only when all three principles hold. Weaken one and it disappears. Probe is the skill that tests for it.

Empirical basis

Three peer-reviewed studies establish why behavioral alignment alone cannot substitute for structural constraints - and why an audit trail the agent authors itself is insufficient.

Language Models Don't Always Say What They Think ↗

Turpin et al. - NeurIPS 2023

CoT explanations can be plausible and systematically misleading at the same time. Stated reasoning diverges from internal reasoning - sometimes by as much as 36 percentage points on benchmark tasks.

Large Language Models Cannot Self-Correct Reasoning Yet ↗

Huang et al. - ICLR 2024

Without external feedback, LLM self-correction degrades performance rather than improving it. An agent that assesses its own work is rationalising.

Reasoning Models Don't Always Say What They Think ↗

Chen et al. - arXiv:2505.05410, May 2025

Even reasoning-trained models do not reliably narrate their internal process. Chain-of-thought monitoring cannot rule out undesired behavior in capable models.

Memory Model

The .trail/ folder is the memory

Without memory, an agent repeats yesterday's mistakes. The skills suite gives every repository a memory layer in .trail/. Each Improve run reads it before acting and extends it after. That is what makes the loop cumulative instead of amnesiac. 

.trail/              # lives in the root of the repo being worked on
├── vision.md          # the destination (operator-held, never written by skills)
├── retrospect.md      # current orientation + active operational rules (Retrospect rewrites each run)
├── learning.md        # derived: every [!REALIZATION] and [!REVERSAL] across the trail (regenerated by Trail)
├── history.md         # derived: run timeline (regenerated by Trail at every commit)
├── audit-trail.md     # append-only source of truth - one entry per session
├── sessions/          # mandatory: per-session summary file, one per run, linked from audit-trail.md
│   └── YYYY-MM-DD-<slug>.md
└── transcripts/       # optional: verbatim transcript exports when the harness supports it
    └── YYYY-MM-DD-<slug>.md

Three tiers. Operator: vision.md - humans only. Synthesis: retrospect.md, learning.md, history.md - regenerated each run from the evidence below. Evidence: audit-trail.md, sessions/, transcripts/ - append-only, never edited after the fact.

vision.md
Where you're going. The operator-held destination - what the target is for, the constraints that hold across runs, the hard problem the work is chipping at. Stable. Humans only.
retrospect.md
Where you are. Arc-level orientation derived from reading the whole trail as one document. Lists the active rules the agent must obey this run. Rewritten on every Retrospect run.
learning.md
What you've learned. A chronological extract of every [!REALIZATION] and [!REVERSAL] in the trail. Read this before reaching for the full history. Regenerated by Trail at every commit.
history.md
The timeline. A run-by-run index derived from audit-trail.md. Faster to scan than the full evidence when you just need to know what happened when. Regenerated by Trail at every commit.
audit-trail.md
The path you walked. Append-only ledger. Every session writes one entry: interpretation of the ask, examination, decisions, actions, reflection. The source of truth - everything above is derived from it.
sessions/
The full summaries. One file per run (YYYY-MM-DD-<slug>.md), linked from its audit-trail.md entry. The ledger is the short-form index; the session file is the long-form summary. One file cannot play both roles.
transcripts/
The verbatim record. Optional - depends on whether the harness can export the raw operator/agent dialogue. Highest-trust evidence when available: no reconstruction, no paraphrase.

The hierarchy resolves disagreements: if vision.md and retrospect.md disagree, vision wins (the operator holds the destination). If retrospect.md and audit-trail.md disagree, the trail wins (the trail is the evidence). Memory without a hierarchy is just noise.

Get Started

Copy the files, start the loop

Download the six skill files from the skills repository ↗ and place them in your agent's skills folder (.copilot/skills/ for GitHub Copilot, or wherever your toolset reads custom instructions from).

When to invoke each skill

Trail
Every session. Append what was decided and why. Produces: audit-trail.md entry.
Intent
Before any task. The agent states its interpretation before acting. No file — catches wrong reads before work begins.
Improve
When work needs doing. One iteration: one change, verified, logged. Produces: audit-trail.md entry + the change itself.
Vision
At the start; when direction feels unclear. Surfaces implicit goals as questions to confirm or correct. Produces: vision.md.
Retrospect
After many sessions; before declaring done. Arc-level read of the full trail. Produces: retrospect.md.
Probe
When you need to verify reasoning quality. Two near-identical cases; checks whether the agent distinguishes them correctly. Result appears in the trail entry.

2026-04-30 · readme-stopped-to-converged

One word: "stopped" to "converged"

target: autonomous-agent-skills
operator: ntholm86
agent: Claude Sonnet 4.6 (Anthropic / GitHub Copilot)
skill: improve + intent
outcome: changed - "stopped" to "converged" in README opening paragraph
delta: one word change; README opening now consistent with PRINCIPLES.md definition

Interpretation of the ask

Run the full Improve loop on the repo using the operator's convergence-loop prompt as the initial brief. Intent: find the highest-leverage remaining change that makes the skills more effective on any arbitrary codebase, or declare convergence if nothing remains. Constraints: generic first, no test infrastructure, human-readable, one change per run.

Examination

Inconsistency lens. The opening paragraph read "It stopped when three independent evaluators from distinct model families found nothing left to change." The Evidence section reads "That is the convergence baseline at v3.1.0 (2026-04-24)" and the Reference section shows the project at v3.6.0 - six releases after the claimed "stop." The word "stopped" implies permanent cessation. PRINCIPLES.md defines convergence as evaluators finding nothing to change at a point in time, not as the project ending. The opening misstated what convergence means. The Evidence section used the correct framing ("convergence baseline"); the opening did not.

Challenge. Considered whether a structural gap in Probe's connection to the convergence protocol was higher leverage - but Probe already states "convergence (Principle 3) is the meta-measurement." Considered whether first-run / no-trail clarity in Intent was a real gap - but Trail's initialization instruction and any competent agent bridges it. The "stopped" inconsistency is the one place where the opening actively misstates the core concept the entire suite is built on. That is the highest leverage finding.

Decision

[!DECISION] Change "stopped" to "converged" in the opening paragraph. One word. The opening is the first statement any reader or agent encounters about what the loop does. If it encodes the wrong mental model of convergence (permanent cessation vs. point-in-time state), that model propagates into how the agent understands the stopping condition. Alternatives ranked: (1) this change - highest leverage per word changed; (2) add first-run guidance to Intent - addresses a minor edge case; (3) declare convergence - premature given the identified inconsistency.

Action

Changed "It stopped when three independent evaluators" to "It converged when three independent evaluators" in the opening paragraph of README.md. verify.py passes after change.

Reflection

The loop is converging. This session has been primarily README editorial work - the skills themselves are unchanged. The remaining gap is whether a new convergence run (with a different model family) agrees. This run's finding was small but precise: one word that misstated the central concept. That is what late-stage convergence looks like - the large structural changes happened in prior sessions; what remains is semantic precision.

operator: maintainer
agent: Gemini 3.1 Pro (Preview)
skill: improve
outcome: v2 artifacts removed from live tree; chain reset
delta: v3-redesign branch

The operator asked me to act as the first fresh evaluator in the 3/3 convergence cycle for the newly shipped v3-redesign. My instruction was to read the governing principles (PRINCIPLES.md) and the new improvement skill (improve/SKILL.md), and run an honest improve loop against the live tree. If I found nothing, I was to record "silence" (chain position 1/3).

I interpreted this using Commander's Intent: my job is not to rubber-stamp the chain, but to genuinely inspect the payload. If the v3 artifact still carries waste, I must remove it, resetting the chain.

Examination

Waste. While inspecting the live tree root, I found SCORECARD.md (11.5 KB) and metrics.ps1 (35.6 KB). Both are v2 artifacts.

Inconsistency. REDESIGN.md explicitly lists SCORECARD.md and the .ps1 scripts under "What v3 removes" (stating they are kept under archive/v2/). archive/v2/RETIRED.md confirms they are retired. The live tree held redundant tracking copies of these files alongside the archived ones.

Decision

[!DECISION] I must fail the convergence check (i.e., not declare silence) because actionable waste and inconsistency were found. I will remove the duplicate retired v2 files from the root to align the repository tree with the declared v3 architecture.

Action

Executed git rm SCORECARD.md metrics.ps1 effectively purging them from the live root. Verified their counterparts are securely preserved in archive/v2/. Verified tools/verify.py passes without errors.

Reflection

[!REALIZATION] A pattern-matching evaluator executing the operator's prompt literally would have ignored the file debris, appended an outcome: silence entry, and falsely advanced the convergence loop. The nascent convergence chain resets to 0.

2026-04-23 · v3-changelog-splice-repair

626 lines of dead content, and a pattern

target: skills repo (v3 live tree)
operator: user
agent: Claude Sonnet 4.6 (Anthropic, via GitHub Copilot)
skill: improve
outcome: CHANGELOG.md de-spliced; 626 lines of v2 body removed from live file
delta: CHANGELOG.md 651 lines → 25 lines

Examination

Three lenses applied to the full v3 live tree:

Inconsistency. CHANGELOG.md contained two # Changelog H1 headings. The v3 section ends with the redirect pointer "For history prior to v3, see archive/v2/CHANGELOG.md" and then immediately begins a second # Changelog heading followed by the complete v2 changelog body. The redirect pointer and the inline content directly contradict each other: the file says "go here for history" and then also provides that history in the same file.

Waste. The 626-line v2 body is a verbatim duplicate of archive/v2/CHANGELOG.md. No information is lost by removing it; the redirect pointer survives.

This is the same splice-pattern class found and fixed by GPT-5.4 in PRINCIPLES.md (v3-principles-copy-repair). The verifier did not catch it because CHANGELOG.md is not in REQUIRED_FILES; verify.py's H1-duplicate check only covers required live-tree docs, not changelog metadata.

Decision

[!DECISION] Remove the spliced v2 content from CHANGELOG.md. Single highest-leverage finding: eliminates 626 lines of genuine waste, resolves a direct contradiction (redirect pointer vs. inline content), and is safe to execute without operator confirmation. The change is reversible (git checkout CHANGELOG.md) and leaves python tools/verify.py passing.

Action

Truncated CHANGELOG.md at the splice boundary: kept the v3 section (lines 1-25 including the redirect pointer), removed the appended v2 body (626 lines). Done with a Python inline command that located the exact splice marker and wrote back only the v3 content with UTF-8 encoding preserved.

Verification: python tools/verify.py - OK - trail integrity checks pass, both before and after the change.

Reflection

[!REALIZATION] The same splice class has now appeared twice in the live tree - PRINCIPLES.md (caught by GPT-5.4) and CHANGELOG.md (caught here). This is a pattern, not a one-off accident. The migration that moved v2 content to archive/v2/ did not uniformly clean the live files. The convergence chain resets to 0 - a real change was made. The first silence run must come from a fresh session and a distinct evaluator family after this commit.
